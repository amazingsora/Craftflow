# Craftflow 改進評估報告

> 評估日期：2026-06-04　評估範圍：backend/app（~7,300 行）、frontend/src（~5,700 行）、tools/Craftflow（legacy CLI）、doc/
> 評估視角：① 軟體開發人員　② 繪圖新手（最終使用者）　③ AI 繪圖／架構專家

---

## 一、總體印象

Craftflow 是一個架構清晰、文件紀律良好的本機優先（local-first）創作輔助系統。後端分層（api / services / models / schemas / core）標準、Prompt Compiler 的中文→SD tag 轉換很有巧思、開發記錄詳實、Code Review 文化成熟（昨日剛完成 10 項問題盤點與修復）。

主要風險不在「功能不足」，而在**規模化後的可維護性**與**對非工程背景使用者的友善度**。以下依三個視角分述，並在最後附上優先順序總表。

---

## 二、軟體開發人員視角

### 2.1 [高] async handler 中同步阻塞 event loop

多數 `ai_text.py`、`ai_art.py` 的路由是 `async def`，卻直接呼叫同步的 `ollama_client.generate()` / `analyze_image()`（內部是 `requests.post`，逾時 300 秒）。只有 `art_generate.py` 用了 `run_in_threadpool`。

**影響**：任一 LLM/視覺請求進行時（可能數十秒），整個 FastAPI event loop 被卡住，其他 API（角色列表、設定頁、健康檢查）全部等待。昨日已替 `_detect_body_coverage` 修過同一問題，但 `ai_text.py` / `ai_art.py` 全線仍未處理。

**根因**：`requests` 為同步函式庫，放進 async 端點等於在事件迴圈上做阻塞 I/O。

**建議**：將所有對 Ollama/ComfyUI 的同步呼叫一律包 `run_in_threadpool(...)`，或把這些端點改回 `def`（FastAPI 會自動丟到 threadpool）。後者改動最小、最符合「surgical update」。

### 2.2 [中] 全域可變狀態不具持久性與執行緒安全

`core/state.py` 用模組層 global 保存 active checkpoint / workflow / lora；`vram_manager.py` 的 `VRAMGuardian` 是 singleton 但 `_current_owner` 無鎖。

**影響**：① 後端重啟後使用者選的 checkpoint/workflow/LoRA 全部重設；② 兩個並發請求同時 `request_focus` 會競爭，VRAM 擁有者狀態可能錯亂（單人本機情境機率低，但 race 仍存在）。

**建議**：把 active 設定落地到 settings 檔或 DB（一張 key-value 表即可）；`request_focus` 加 `asyncio.Lock`。維持單人使用前提下可列為中優先。

### 2.3 [中] 手刻 migration，缺乏版本化遷移工具

`database.py::_migrate()` 用 `ALTER TABLE ADD COLUMN` 逐欄硬補。能跑，但：無法改型別、無法降級、無版本紀錄、欄位定義與 model 容易漂移。`升級方向.md` 已規劃導入 Enum 與 Outfit/OutfitItem 新表——這類「改型別＋建關聯表」用現行 `_migrate()` 幾乎做不到。

**建議**：導入 **Alembic**。先 `alembic stamp` 既有結構，之後 schema 變更（含 Enum、Outfit 系統）走正規 migration。這是支撐 `升級方向.md` 的前置工程。

### 2.4 [高] 零自動化測試

全專案僅 `tools/Craftflow/test_pro_mode.py` 一支；backend/app 與 frontend 無任何測試。17k 行、且核心 `compiler.py` 充滿 regex 與條件分支（防幻覺、異色瞳、特徵注入），極易在後續修改中回歸。

**建議**：先針對「純函式、無外部依賴、價值最高」的部分補單元測試——`prompt_engine/compiler.py`（`_extract_color_anchors`、`_inject_heterochromia`、`_clean_clothing_hallucinations`、`_reorder_tags`）與 `lexicon` 對照表。這些不需要 Ollama/ComfyUI 即可測，CP 值最高。

### 2.5 [中] 巨型檔案，職責過載

`art_generate.py` 2,256 行、`CharacterTab.jsx` 2,388 行。前者混合了 workflow 載入、IPA/CN 注入、bypass、prompt 注入、seed 處理、coverage 偵測、角色與變體生圖等多責任；後者 250 處 inline style。

**影響**：閱讀成本高、Code Review 困難、衝突率高、無法局部測試。

**建議**：不必大重構（違反專案原則），但可低風險拆檔：把 `art_generate.py` 的 workflow 圖操作（inject/bypass/resolve_conditioning）抽到 `services/ai/workflow_graph.py`，端點只留協調邏輯。前端把共用 style 物件抽成 `styles.js` 或改用 CSS class。

### 2.6 [中] 錯誤以「字串」回傳，契約脆弱

`ollama_client` 不丟例外，而是回傳 `"[Ollama unavailable at ...]"` 字串；呼叫端靠 `raw_response.startswith("[")` 判斷失敗（見 `compiler.compile`）。

**影響**：一旦正常輸出剛好以 `[` 開頭（例如模型回了 `[smile], ...`），會被誤判為錯誤；反之錯誤字串也可能被當成 prompt 拼進結果。型別系統完全無法協助。

**建議**：改用例外或回傳 `Result` 物件（`ok: bool, value/error`）。至少把哨兵字串換成不可能與正常輸出衝突的結構。

### 2.7 [低] 其他

- `requirements.txt` 全用 `>=` 未鎖版本，無 lockfile → 建置不可重現。建議產出 `requirements.lock` 或改用 `uv`/`pip-tools` 釘住版本。
- `main.py` 全域 exception handler 回傳 `type(exc).__name__: exc` 給前端，洩漏內部細節。本機可接受，但與 `CORS allow_origins=["*"]`（已標記 Phase 4 收斂）一樣，對外開放前務必處理。
- 缺 `README`：新機器／協作者上手只能讀散落的 setup 文件。建議補一份「啟動三步驟」README 指向既有 `setup_guide.md`。

## 三、繪圖新手（使用者）視角

### 3.1 [高] 介面術語門檻過高

對只想「畫出腦中角色」的新手，目前 UI 直接暴露 CN weight、IPA、ControlNet、LoRA、checkpoint、workflow JSON 等專業概念。`0.85 CN 權重`、`bust=0.3` 這些數字對新手沒有意義。

**建議**：提供「**簡易模式 / 進階模式**」切換。簡易模式只留三件事：選角色、丟參考圖、按生成；把權重、workflow、negative 等收進「進階」摺疊區並給好預設。用白話標籤取代術語（例：CN 權重 → 「跟參考圖的相似度：寬鬆 / 適中 / 嚴格」）。

### 3.2 [高] 錯誤訊息是給工程師看的

目前失敗會直接顯示後端原字串，如 `[Ollama unavailable at http://host.docker.internal:11434: run \`ollama serve\` and ensure OLLAMA_HOST=0.0.0.0]`。新手看到只會困惑。

**建議**：前端做訊息映射層，把技術錯誤轉成可行動的中文提示，例如「AI 文字服務尚未啟動 → 請先開啟 Ollama，或點此查看教學」。並在頁面頂部用狀態燈顯示 Ollama / ComfyUI 是否在線（`is_available()` 已有，接上即可）。

### 3.3 [高] ComfyUI 工作流匯出陷阱

`2026-06-04 開發記錄` 的 `AnimaStandardV5.json` 事件：使用者把 UI 格式（非 API format）的 workflow 放進 `custom_workflows`，導致永遠 422、完全無法使用，且錯誤本身不解釋原因。這對新手是致命的隱形地雷。

**建議**：上傳 workflow 時即時驗證格式，偵測到 UI 格式（含 `nodes` 陣列）就明確提示「這是 UI 格式，請在 ComfyUI 用『Save (API Format)』重新匯出」，附圖示步驟。能自動轉換更好，但至少先擋下並說人話。

### 3.4 [中] 長時間等待缺乏掌控感

單次角色生圖實測 ~57 秒。已新增逐項耗時 UI（很好），但建議補：① **取消按鈕**（長任務可中止，避免乾等）；② 進度條期望值（「通常約 1 分鐘」）降低焦慮。

### 3.5 [中] 缺少以角色／專案為單位的成品藝廊

目前歷史紀錄存在 localStorage（上限 100、縮圖 jpeg），對寫小說、需要長期累積角色立繪的使用者，找圖會越來越亂。

**建議**：提供以「專案 → 角色」分層的圖庫頁，可篩選、標記主視覺（這也呼應 `升級方向.md` 提的 main_illustrations 多對多）。

### 3.6 值得肯定

中文→prompt 自動編譯（新手不必懂英文 tag）、逐步進度回饋、深色 UI 與中文分頁標籤、防幻覺自動修正——這些都實質降低了新手門檻，方向正確。

## 四、AI 繪圖／架構專家視角

### 4.1 [高] 防幻覺邏輯靠硬編碼關鍵字，不可擴展

`compiler.py` 的 `_clean_clothing_hallucinations`、`_inject_traits`、`_CONTEXT_BLOCKERS`（大小姐→封鎖女僕）、異色瞳處理，全是寫死的中文關鍵字清單與 if 分支。巧妙，但每加一種特徵就要改程式碼，且 regex 跑在自由文本上易誤判。

**建議**：走**資料驅動**。把「觸發詞 → 強制 tag / 封鎖 tag / 注入 negative」抽成設定檔（YAML/JSON 或 DB 表），程式只讀規則。更根本的方向是讓角色屬性**結構化**（髮色、瞳色、服裝部件各自獨立欄位，正是 `升級方向.md` 的 Outfit/OutfitItem 思路），prompt 直接由結構化欄位組裝，而非對一段中文描述做正則考古——可同時解決幻覺與可維護性。

### 4.2 [高] 「半身參考圖→全身輸出」用 Canny 是方法上的死路

開發記錄已誠實指出核心矛盾：Canny CN 高權重會精確複製半身構圖、低權重又風格飄移。目前的「依 coverage 動態降權」是合理的止血，但治標。

**建議**：**用對的 ControlNet 類型解耦「結構」與「風格」**。
- 結構改用 **OpenPose / DWPose**（骨架）或 **Depth**，而非 Canny。給一副全身骨架，模型就會畫全身，不再被半身邊緣鎖死。
- 風格／identity 維持由 **IPAdapter** 負責。
- 真要從半身延伸全身，正解是 **outpainting（向下擴圖）** 或先用 pose 骨架生全身再 i2i，而非調 Canny 權重。

### 4.2b [中] 用視覺 LLM 做姿態偵測過於昂貴

`_detect_body_coverage` 呼叫 `qwen2.5vl` 判斷全身/半身/胸上，實測 ~10 秒，且結論本身不穩。ComfyUI 生態已有 **DWPose/OpenPose preprocessor**，可直接回傳關鍵點，判斷可見部位又快又準。

**建議**：以 pose preprocessor 取代視覺 LLM 做 coverage 判定，省下約 10s 並提高可靠度。

### 4.3 [高] 角色跨圖一致性，IPAdapter 不夠，應接上既有 LoRA 訓練

寫小說需要同一角色在多張插圖維持一致。單靠 IPAdapter 會有 identity 漂移。專案**已經有完整 LoRA 訓練基礎設施**（`lora_trainer/`、`training.py`、`TrainingTab`），但生圖端點直到 2026-06-04 才剛把 global LoRA 接進角色生圖。

**建議**：把「**每角色一顆 LoRA + 觸發詞**」做成主線流程——角色模型關聯其專屬 LoRA，生圖時自動掛上。這是角色一致性最穩的解，且你已具備全部零件，只差把流程串起來並在 UI 暴露「為此角色訓練專屬模型」入口。

### 4.4 [中] VRAM 在 Ollama 與 ComfyUI 間反覆換載，造成 thrashing

單次角色生圖流程是：視覺分析（Ollama）→ 姿態偵測（Ollama）→ prompt 編譯（Ollama）→ 生圖（ComfyUI）。`VRAMGuardian` 每次切換都 unload 對方、下次再 reload。若中間穿插，等於反覆吃模型載入時間。

**建議**：**批次化**——把所有 LLM 工作（視覺＋偵測＋編譯）一次跑完再交棒給 ComfyUI，全程只切一次 focus。實作上把三個 Ollama 步驟聚在一起、最後才 `request_focus("comfyui")`。先用現有 timing 量化切換成本再決定力度。

### 4.5 [中] 可重現性：生成參數要完整存檔

已正確處理 seed（`-1` 自動補隨機、回填到 KSampler/noise_seed），這很好。但專家需要的是**整組參數可回溯**：seed + 正/負 prompt + checkpoint + 每顆 LoRA 名稱與權重 + CN 類型與權重 + sampler/steps/cfg。

**建議**：每張成品連同上述完整 metadata 一起存（DB 或寫進 PNG 的 info chunk）。如此「重現某張滿意的圖」「微調再生」才可行，也利於日後 debug。

### 4.6 [中] ComfyUI synthetic UI-workflow shim 脆弱

`comfyui_client.submit_workflow` 為了讓某些 custom node（KJNodes、Impact Pack 等）不崩，反推了一個假的 UI 格式 `nodes` 結構塞進 `extra_pnginfo`。這是逆向 ComfyUI 內部慣例，ComfyUI 一改版就可能失效。

**建議**：中長期應傾向「白名單已驗證 workflow」並記錄相容的 ComfyUI 版本；shim 視為相容層、加註解與版本守備。短期維持即可，但別在其上疊更多 hack。

### 4.7 [中] 生圖前未預檢模型／檔案是否存在

開發記錄 #9、#10 已自承：IPA/CN 注入時硬編碼模型檔名、未預檢 ComfyUI 是否有該檔；bypass 安全檢查也有未覆蓋路徑。新手換機器或少裝一個模型，會得到難解的 ComfyUI 錯誤。

**建議**：生圖前用 `/object_info` 與模型清單做 preflight check，缺檔就回傳明確「缺少 X 模型，請安裝」訊息（呼應 3.2 的友善錯誤）。

## 六、一句話結論

**工程面**先解 async 阻塞與測試空窗；**使用者面**先把錯誤訊息與術語講成人話；**繪圖品質面**最關鍵的單一改動，是把 ControlNet 從 Canny 換成 OpenPose/Depth、並把既有的 LoRA 訓練接成「每角色專屬模型」主線——這兩件事能同時根治目前最痛的「半身參考圖畫不出全身」與「角色跨圖不一致」。
