## 問題清單（明日開發目標）

### 問題 1：切換到「checkpoint 模式」後 CN / IPA 不套用 ✅

**現象**：使用者有一支含 CN + IPA 節點的自訂工作流（`Standard_V35.json`）。切換到 **checkpoint 模式**（UI 右上角的模式切換）後，角色生圖的 CN / IPA 全部消失不套用。

**根因（已確認）**：`App.jsx` 定義了兩種生成模式：
- **checkpoint 模式**（預設）：`onGenerationModeChange('checkpoint')` 時，前端把後端 workflow 同步為 `text_to_image.json`（系統工作流，**無 CN/IPA 節點**）。
- **workflow 模式**：後端 workflow 才設為使用者選定的自訂工作流（`Standard_V35.json`，**有 CN/IPA 節點**）。

角色生圖端點（`/art/characters/{id}/generate-design`）直接讀 `state.get_workflow()`（即 `active_wf`）。切換到 checkpoint 模式後，`active_wf = 'text_to_image.json'`，無 IPA 節點 → `_wf_has_ipa()` 回傳 False → IPA / CN 全部 bypass。

**確認設計（2026-06-01）**：

角色生圖端點改為「角色生圖永遠使用有 IPA 的工作流，並強制套用全域 checkpoint」：
1. 讀 `state.get_workflow()`，若無 IPA 節點 → 掃 `CUSTOM_WORKFLOWS_DIR/` 找第一個含 IPA 的 JSON（預期命中 `Standard_V35.json`）。
2. 找到後，不管是 custom 還是 system workflow，強制把全域 checkpoint 注入 `CheckpointLoaderSimple.ckpt_name`。
3. `_load_workflow` 新增 `force_checkpoint: bool = False` 參數，傳 `True` 時繞過「custom 不覆寫」限制。

改動範圍：`art_generate.py` 角色生圖端點開頭 ~15 行 + `_load_workflow` +3 行。**低風險。**

### 問題 2：工作流未含 CN / IPA 節點 → 懶人化自動檢測 ✅

**現象**：若使用自訂 workflow（非 Standard_V35），而該 workflow 沒有 CN / IPA 節點，送出後直接失敗或靜默跳過，使用者不知如何處理。

**需求**：不要求使用者手動編輯 workflow JSON，需要系統自動處理。

**確認設計（2026-06-01）**：做「自動注入」。

節點注入點：找到 `KSampler`（取 `denoise=1.0` 的那個，避免 hires-fix 誤命中）→
- `model` 現有來源 → 插入 IPA 鏈後重接
- `positive` / `negative` 現有來源 → 插入 CN Apply 後重接

注入模板（模型檔名來自 Standard_V35.json，已確認存在）：

| 節點 | 關鍵設定 |
|------|---------|
| `CLIPVisionLoader` | `CLIP-ViT-H-14-laion2B-s32B-b79K.safetensors` |
| `IPAdapterModelLoader` | `ip-adapter-plus_sdxl_vit-h.safetensors` |
| `IPAdapterAdvanced` | weight 讀前端傳入值；接 LoadImage(IPA ref) |
| `ControlNetLoader` | `diffusion_pytorch_model_promax.safetensors` |
| `SetUnionControlNetType` | type = canny |
| `CannyEdgePreprocessor` | 接 LoadImage(CN ref) |
| `ControlNetApplyAdvanced` | strength 讀前端傳入值；start=0.0, end=0.85 |

節點 ID：`max(int(k) for k in wf if k.isdigit()) + 1` 開始遞增，確保不衝突。

新增函式：`_inject_ipa_cn_nodes(wf, ipa_weight, cn_weight)` → 回傳 `(ipa_loadimage_id, cn_loadimage_id)` 供後續填入圖片。

風險：多 KSampler 工作流（Hires-fix）需用 denoise 判斷主路徑；單 KSampler 工作流零風險。

### 問題 3：男性角色套用效果落差大 🔲

**現象**：女性角色 CN + IPA 效果良好；男性角色外觀相符度明顯不足（輪廓、服裝、細節偏差大）。

**推測根因**：
- 訓練資料偏斜：SDXL / 常用 checkpoint 女性人物資料遠多於男性。
- Prompt 缺乏男性強化 tag（如 `male, masculine, broad shoulders, flat chest`）。
- CN 的 Canny 邊緣對男性服裝結構（如西裝、盔甲）與女性截然不同，命中率較低。

**明日調查點**：
1. 後端確認男性角色生圖時是否自動注入男性強化 positive tag（目前僅靠角色描述）。
2. 考慮在 `art_generate.py` 對 `character.gender == 'male'` 追加 `male focus, masculine features, 1boy` 等保底 tag。
3. IPA weight 對男性是否需要調高（女性 0.15 可行，男性可能需 0.2–0.3）。

### 問題 4：多草圖合成 — 主圖 + 設定圖合併為 CN 輸入 🔲

**需求說明**：
- 目前可上傳最多 3 張草圖，全部用相同方式處理。
- 新邏輯：
  - **草圖 1（主圖）**：作為 IPA 外觀參考（現行行為，保留）。
  - **草圖 2 + 3（設定圖）**：合併成一張圖作為 CN 結構/姿勢輸入。
- 使用者觀察：CN 強度 ≥ 0.9 比 IPA 更有用，畫風差異在可接受範圍內。

**技術評估（初步可行）**：
- 合併方式：左右並排（side-by-side）或上下並排，再 letterbox 至畫布比例。
- 後端 `_letterbox_to_aspect` 已有實作基礎可延伸。
- IPA 繼續用草圖 1（原始）；CN 改用合併後的草圖 2+3。
- 若只上傳 1 張草圖 → 維持現行：草圖 1 同時作 IPA + CN 輸入（fallback 不破壞現行行為）。
- 若只上傳 2 張 → 草圖 1 IPA；草圖 2 CN（單張，不合併）。

**風險**：
- 兩張設定圖並排後人物縮小（各佔寬度一半），Canny 擷取的細節變少。
- 可能需要先 resize 各自至標準高度再拼接，確保細節保留。

**前端改動**：
- CharacterTab 需顯示「草圖 1：外觀主圖 / 草圖 2–3：結構參考」標示，讓使用者知道用途差異。

## 實作記錄：問題 1 + 問題 2 合併解法（統一注入管線）

### 決策

捨棄問題 1 原方案（checkpoint 模式時「去掃別的工作流借一個有 IPA 的來用」），改用**統一注入管線**一次解決問題 1、2。使用者需求三點本質為同一邏輯：

1. checkpoint 模式開 IPA/CN → 注入（`text_to_image.json` 無節點）
2. 工作流模式開 IPA/CN → 缺節點才注入
3. 工作流已有 IPA/CN → 沿用既有、不重複注入

統一為：**「啟用且有參考圖 → 缺節點則建、有則沿用；停用或無圖 → 既有節點 bypass」**。比借工作流更乾淨，且不會有「借錯工作流」風險。

### 改動（`backend/app/api/art_generate.py`）

1. **新增 `_find_main_ksampler_id(wf)`**：單一 KSampler 直接回傳；多 KSampler（hires-fix）取 `denoise==1.0` 主路徑，避免命中二次取樣。
2. **新增 `_inject_ipa_cn_nodes(wf, *, inject_ipa, inject_cn)`**：依 Standard_V35 模板動態建立節點並接線。
   - IPA：在「現有 model 來源 → KSampler.model」間插入 IPAdapterAdvanced 鏈（CLIPVisionLoader / IPAdapterModelLoader / LoadImage / IPAdapterAdvanced）。
   - CN：在「現有 positive/negative → KSampler」間插入 ControlNetApplyAdvanced 鏈（ControlNetLoader / SetUnionControlNetType / CannyEdgePreprocessor / LoadImage / ControlNetApplyAdvanced）。
   - 節點 id 從 `max(數字 id)+1` 遞增；模型檔名沿用 Standard_V35（已確認存在）並抽成模組常數，無 magic string。
3. **改寫兩端點生成段**（`generate-design`、`variants/{slot}/generate-design`）：把原「有 IPA/無 IPA 兩條分支」改為先 `_load_workflow` → 算 `need_ipa_inject` / `need_cn_inject`（功能啟用且有圖且工作流缺該節點）→ 缺則注入 → 既有 bypass 邏輯不變。圖片注入、權重設定沿用既有 `_inject_ipa_image` / `_inject_controlnet_image`（已驗證能命中新建的 LoadImage 與 CannyEdgePreprocessor）。
   - 順手移除原 `[CRAFTFLOW-DEBUG-CANDIDATE/ELSE/AFTER-LOAD]` 暫時除錯 print。

### 驗證

獨立腳本模擬注入（`text_to_image.json` / `Standard_V35.json`），四情境全過、無懸空 link：
- t2i +IPA+CN：`KSampler.model←IPA`、`positive/negative←CN`，CN 接回原 CLIPTextEncode。
- t2i 僅 IPA / 僅 CN：另一側維持原接線。
- Standard_V35（已含兩者）：`need_inject` 皆 False，不重複注入。

> ⚠️ 待 ComfyUI 實機驗證：CLIP-ViT-H / ip-adapter-plus / promax 三個模型檔需確實存在；建議首次以 checkpoint 模式 + 開 IPA+CN 跑一張全身圖確認。
> 註：本次 sandbox 的 Linux mount 對該檔讀取有同步延遲，`py_compile` 無法對最新檔內容執行；改動區塊已逐段審閱，邏輯區塊獨立測試通過。
