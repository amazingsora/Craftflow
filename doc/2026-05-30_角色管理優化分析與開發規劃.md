> 目的：針對五個痛點先做根因分析，明天據此開發。

## 痛點 1：IPA 與 ControlNet 共用同一介面控制強度

### 現狀
- 後端其實是**分離**的兩個參數：`generate_character_design()` 接受 `ipa_weight`（→ `IPAdapterAdvanced.weight`）與 `cn_weight`（→ `ControlNetApplyAdvanced.strength`）。見 `art_generate.py:962-974`、`art_generate.py:1216-1219`。
- 問題出在**前端**。`CharacterTab.jsx:1168-1171`（主角色）與 `1376-1379`（變體）：
  ```js
  use_ipa: ipaEnabled ? '1' : '0',
  ipa_weight: String(ipaWeight),
  use_controlnet: ipaEnabled ? '1' : '0',   // ← 直接綁 ipaEnabled
  cn_weight: '0.7',                          // ← 寫死，無 UI
  ```
- UI 上只有一個「IPA」開關（`ipaEnabled`）與一條「IPA 強度」slider（`ipaWeight`，`CharacterTab.jsx:1561-1585`）。**沒有任何 ControlNet 的開關或強度 UI**。

### 根因
前端把 `use_controlnet` 綁在 `ipaEnabled` 同一個 checkbox，且 `cn_weight` 寫死 0.7。所以使用者操作上「一個開關同時開關兩者、一條 slider 看起來像在控制全部」，實際 slider 只動 IPA、ControlNet 強度永遠 0.7。

### 建議方案（前端為主，後端已支援）
1. 在 `CharacterTab.jsx` 把 IPA 與 ControlNet 拆成**兩個獨立 checkbox + 兩條 slider**：新增 `cnEnabled`、`cnWeight` state（比照 `ipaEnabled`/`ipaWeight` 的 `_loadGenPrefs`/`_saveGenPref` 寫法，含變體 `vState`）。
2. 送出時改為 `use_controlnet: cnEnabled ? '1' : '0'`、`cn_weight: String(cnWeight)`。
3. 變體分支（`1376-1379`、UI `1978-1999`）同步處理。

### 風險
- `_loadGenPrefs` 預設值需補 `cnEnabled`/`cnWeight`，避免舊偏好讀取為 undefined（`CharacterTab.jsx:1002-1012`）。
- 改動集中在前端 state 與 form 組裝，後端不需動，風險低。

## 痛點 2：必須同時設定好 IPA 與 ControlNet 節點才能用，不能拆開

### 現狀
- 角色生成走 `state.get_workflow()` 指定的 active workflow。目前同時含 IPA + ControlNet 的工作流是 `data/custom_workflows/Standard_V35.json`（節點：`IPAdapterModelLoader` + `IPAdapterAdvanced` + `ControlNetLoader` + `SetUnionControlNetType` + `CannyEdgePreprocessor` + `ControlNetApplyAdvanced`，全部串在 `KSampler` 之前）。
- 後端有 `_bypass_ipa_nodes()`（`art_generate.py:486`）：無 IPA 參考圖時會把 IPA 節點移除並重接 `upstream_model → KSampler`。
- **但沒有對應的 `_bypass_controlnet_nodes()`**。當不想用 ControlNet 時，CN 節點仍留在工作流裡執行，只是不注入圖、不改 strength（`art_generate.py:1218-1219`、`1226-1230` 只在 `_cn_ref_bytes` 非 None 時才注入）。

### 根因
ControlNet 缺少「停用即從工作流剝離」的繞過邏輯，與 IPA 不對稱。因此一旦 active workflow 內嵌 CN 節點，就無法單獨關閉 CN 而只跑 IPA（反之 IPA 可被 bypass）。「兩者綁死」即源於此。

### 建議方案（後端）
1. 在 `art_generate.py` 新增 `_bypass_controlnet_nodes(wf)`，比照 `_bypass_ipa_nodes` 的重接線思路：找到 `ControlNetApplyAdvanced`，把它的 `positive`/`negative`（conditioning 輸出）還原接回上游 conditioning，再移除 CN 相關節點（`ControlNetApplyAdvanced`、`ControlNetLoader`、`SetUnionControlNetType`、preprocessor、CN 專用 `LoadImage`）。
   - 注意 `ControlNetApplyAdvanced` 是雙輸出（positive/negative conditioning），重接線比 IPA（單一 model 輸出）複雜，需分別處理兩條 conditioning。
2. 在主/變體生成流程中，當 `use_controlnet` 為 False 或無 `_cn_ref_bytes` 時呼叫 bypass（對齊現有 `art_generate.py:1196-1202` 的 IPA 處理位置）。

### 風險
- 重接 conditioning 線若處理錯會導致 prompt 不生效或 ComfyUI 422。**務必先用 `Standard_V35.json` 實測**，並用 `_log_wf_snapshot` 驗證 bypass 後節點結構。
- 替代低風險方案：另存純 IPA 與純 ControlNet 兩個 workflow JSON，讓使用者在 settings 選擇；不動重接線邏輯。可作為明天的保底方案。

## 痛點 3：希望根據上傳圖「全身化」並補足缺漏（如手指），但目前只生成中段（下巴到大腿）

### 現狀
- 全身模式 prompt 後綴已含 `full body, front view, character design sheet`，尺寸 `768×1280`（`art_generate.py:1126-1132`）。
- 全身模式仍**同時啟用 ControlNet**，且 IPA 與 CN 共用同一張原始概念圖（`_ipa_ref_bytes = _cn_ref_bytes = valid_images[0]`，`art_generate.py:1034-1039`）。
- ControlNet 走 `CannyEdgePreprocessor`（Standard_V35）。

### 根因（兩層）
1. **構圖被 ControlNet 鎖死**：Canny 取原圖邊緣作結構約束。若上傳的概念圖本身只有「下巴到大腿」的中段構圖，CN 會強制輸出跟隨該 framing，壓過 prompt 的 `full body`。IPA weight 0.6 也會額外帶入參考圖的構圖傾向。這是「只生成中段」的主因。
2. **缺漏部位無防護**：SDXL／Pony／Illustrious 的預設 negative **不含** `missing fingers / bad hands / cropped`（見 `styles.py:234-237`、`243`、`264`；只有 NOOBAI、AnythingXL 有，`styles.py:255-258`、`271-275`）。正向也沒有手部強化標籤。所以未畫的手指不會被主動補全。

### 建議方案
1. **全身模式解除構圖鎖定**（擇一或組合）：
   - 全身模式時停用 ControlNet（依賴痛點 2 的 bypass），改由 IPA 只傳遞外觀/風格、不鎖姿勢；或
   - 將全身模式的 CN 預處理由 Canny 改為 **OpenPose 全身骨架**，並餵入標準全身姿勢圖而非使用者的中段原圖；或
   - 顯著調低全身模式 `cn_weight`（如 0.3–0.4）並降低 `ipa_weight`。
2. **強化 framing 與補全標籤**（`art_generate.py:1127-1131` suffix）：
   - 正向補：`full body shot, head to toe, standing, full body visible, detailed hands, five fingers`。
   - 全身模式專屬 negative 補：`cropped, out of frame, cut off, close-up, portrait, missing fingers, extra digits, bad hands, fused fingers`。
   - 建議用具名常數集中管理，避免 magic string 散落（對齊 CLAUDE.md「Avoid magic numbers / 可重用」）。
3. **尺寸**：`768×1280`（比例 0.6）對部分 SDXL 底模偏離訓練分佈，可評估改 `832×1216`（SDXL 原生縱向比）以穩定全身構圖。

### 風險
- 停用/弱化 ControlNet 會降低與原圖的「形似」，但對「全身化」是必要取捨——符合 Human-in-the-Loop（產出供參考，不覆寫原作）。
- 改 OpenPose 需 ComfyUI 已裝對應 preprocessor 與 controlnet 模型，明天需先確認環境。

## 痛點 4：程式碼深度分析

### 生成流程鏈
`generate_character_design`（`art_generate.py:962`）：
1. 解析 art_style 優先序（參數 > character > project）。
2. 載入概念圖（最多 3 張，`1021`），`_is_flat_color_draft` 判斷是否純色草稿（`732`）。
3. 視覺分析：`qwen2.5vl` 抽中文視覺特徵（`1045`）。
4. 文字編譯：`compile_prompt` 把中文 → Danbooru tags（`1088`）。
5. 組裝 prompt：gender/age/height prefix + ai_prompt + 主體 + suffix + 個人風格（`1134-1165`）。
6. 工作流注入：LoRA → prompts → 尺寸/seed/steps → IPA weight → CN strength（`1203-1230`）。
7. 提交 ComfyUI、回傳 PNG + 一組 `X-*` debug header。

### 觀察與技術債
- **主角色與變體大量重複**：`generate_character_design`（962-1257）與 `generate_variant_design`（1262-1515）約 80% 邏輯重複。建議抽共用函式（prompt 組裝 + 工作流注入），但屬重構，需符合「不必要不重構」——可列為後續，明天先不動。
- **遺留 debug print**：`art_generate.py:980, 1185, 1198, 1202, 1235-1236` 多處 `print("[CRAFTFLOW-DEBUG...]")` 直寫 stdout。建議收斂為 `logger.debug`，標 TEMP 的應移除。
- **IPA/CN 共用同一參考圖**寫死（`1034-1039`）：未來若要分別餵圖（IPA 給外觀圖、CN 給姿勢圖）需解耦。
- **錯誤韌性大致良好**：ComfyUI/Ollama 失敗都轉 HTTPException（`282-298`、`_run`），符合 CLAUDE.md「Resilient errors」。但 `_inject_controlnet_image` 找不到節點只 warning 不報錯（`585-587`），使用者會以為有用到 CN 卻沒有——建議在 `X-CN-Used` 已有回報，前端應顯示該狀態（目前前端只顯示 IPA on/off，`CharacterTab.jsx:1872-1877`，可補 CN 狀態 chip）。
- **`_replace_negative_seeds` 與 KSampler seed 設定可能重複**（`405-408`、`1213-1214` 後又呼叫 `1232`），需確認無覆蓋衝突。

## 痛點 5：語言模型深度分析

### 模型與職責
- `dolphin-llama3`（文字）：`compile_prompt` 中文 → Danbooru tags、角色摘要/抽取/設計問答。
- `qwen2.5vl:7b`（視覺）：概念圖 → 中文視覺特徵描述。

### 編譯管線（`prompt_engine/compiler.py:164`）
LLM 翻譯 → `_extract_output`（抓 `Output:` 後段）→ `_sanitize_to_list`（去 banned/重複）→ 防幻覺：服飾（`_clean_clothing_hallucinations`）、語義特徵注入（`_inject_traits`，含大小姐→阻女僕）、情緒召回（笑→smile）、異色瞳（`_inject_heterochromia`）→ anchor 抽取（髮色/眼色，`_extract_color_anchors`）→ 衝突清除（`_remove_conflicting_tags`）→ 重排加權（`_reorder_tags`，anchor 加 `:1.1`）→ 拼 quality_prefix。

### 觀察與風險
- **雙跳翻譯誤差**：視覺(qwen 輸出中文) → 文字(dolphin 再翻英文 tags)，兩段串接累積誤差；視覺已是不確定描述，再經第二模型「翻譯」易走樣。可評估讓 qwen 直接輸出 tag 傾向，或用 WD14（`/art/wd14-tags` 已存在，`art_generate.py:1522`）直接從圖反推 Danbooru tag，繞過中文中介。
- **視覺細節流失**：`ollama_client.py:22-23` 多圖 resize 到 448px、單圖 768px。手指、服裝細節在 448px 幾乎不可辨——與痛點 3「補手指」相關，靠視覺分析救不回，須靠 prompt/negative。
- **輸出長度上限**：vision `num_predict=160`、compile `num_predict=250`（`compiler.py:182`、`art_generate.py:1047`），複雜角色可能截斷。
- **anchor 僅涵蓋髮色/眼色**（`lexicon`），膚色、體型、服裝色彩無確定性錨定，仍依賴 LLM 自由發揮，一致性較弱。
- **temperature**：compile 0.3、vision 0.1，偏穩定，合理。
- **FLUX 不走 tag 清洗**（`compiler.py:193,218`），自然語言路徑；目前角色生成走 tag 系底模，影響不大。
- **防幻覺規則為硬編關鍵字表**（`_CONTEXT_BLOCKERS`、casual_signals），覆蓋有限、需逐案擴充，屬可維護性議題而非錯誤。
