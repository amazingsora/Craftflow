## 技術決策

### `_resolve_conditioning` 設計
ControlNetApplyAdvanced 有兩個 conditioning 輸出：slot 0 = positive conditioned、slot 1 = negative conditioned。
Pass 1 從 KSampler 邊的 `[node_id, output_slot]` 出發，根據 output_slot 遞迴穿透中間節點，確保即使有 CN 注入時也能找到正確的 CLIPTextEncode。

### global LoRA 優先順序
與 `/art/generate` 對齊：global_lora（state） → art_style.loras，順序一致。

## 姿態偵測 → 動態 CN 權重

### 問題
`_shrink_for_full_body` 把半身/胸上參考圖縮小後，CN 0.85 權重仍會精確複製縮小的半身結構，SD 補出的是背景而非腿部，全身生圖無改善。**姿態偵測的 10s 沒有實際效果。**

### 根本矛盾
- CN 高權重（≥0.85）= 嚴格跟結構，半身進 → 半身出
- CN 低權重（<0.6）= 風格飄移

### 修改：依偵測結果動態覆蓋 CN 強度

新增常數 `_COVERAGE_CN_WEIGHT`（`art_generate.py`，緊跟 `_BODY_TOP_OFFSET`）：

```python
_COVERAGE_CN_WEIGHT = {
    "full":    None,   # 全身 → 維持使用者設定（不覆蓋）
    "partial": 0.5,    # 半身 → 降低，讓 SD 補下半身
    "bust":    0.3,    # 胸上 → 大幅降低，近似只靠 IPA 抓風格
}
```

偵測完 coverage 後立即覆蓋 `cn_weight`，在 char-gen 與 variant-gen 各加了 4 行。
`_shrink_for_full_body` 縮圖邏輯保留（低 CN 權重下仍有方向性幫助）。

### 預期行為
- 全身參考圖 → CN 維持 0.85，結構精確
- 半身參考圖 → CN 降到 0.5，SD 根據 positive prompt（full body shot, head to toe...）補全腳部
- 胸上參考圖 → CN 降到 0.3，結構幾乎不受參考圖拘束，IPA 主導風格一致性

### 待確認
- `partial=0.5` / `bust=0.3` 數值需實測調整，視模型對 CN 敏感度而定
- 若效果仍不理想，`bust` 考慮直接 bypass CN（設為 `None` 並跳過 CN 注入）

## 後續優化（承接同日改進評估報告）

依 `doc/2026-06-04_專案改進評估報告.md` 的優先順序，完成兩項低風險、高投報率改善。

### 優化 A：消除 async 端點的 event loop 阻塞（報告 P0 / 2.1）

**根因**：`ai_text.py`、`ai_art.py` 的端點宣告為 `async def`，卻在 `await guardian.request_focus(...)` 之後直接呼叫同步的 LLM／視覺服務（底層 `requests.post`，逾時 300s）。AI 推論的數十秒內，整個 FastAPI event loop 被卡住，其他 API（角色列表、設定頁、health check）一併等待。先前僅 `art_generate.py` 套用 `run_in_threadpool`。

**修法**（比照 `art_generate.py` 既有模式，最小變更、不動架構）：將阻塞的同步呼叫包進 `await run_in_threadpool(...)`。

`ai_text.py`（+1 import，包裝 4 處）
- `consistency_service.analyze`
- `rewrite_service.batch_rewrite`
- `character_service.extract_from_text`
- `character_service.design_chat`

`ai_art.py`（+1 import，包裝 6 處）
- `art_service.critique_sketch` / `critique_finished` / `advise_color`
- `art_service.describe_illustration`
- `art_service.composition_ask`（圖文問答、自由問答各一）
- `character_service.describe_portrait`

**未動**：`rhythm_service.analyze`（純 CPU regex、本機快速、非網路阻塞）刻意保留，維持最小變更。

**風險**：低。回傳值與 API 契約不變，僅將阻塞工作移至 threadpool 執行緒；`db` session 仍留在 async handler、未進入子執行緒，無 SQLAlchemy 跨執行緒問題。

### 優化 B：Workflow UI 格式提前預警（報告 3.3）

**根因**：ComfyUI 以「Save」（非「Save (API format)」）匯出的 workflow 含頂層 `nodes` 陣列，無法送進 `/prompt`。原本只有 `art_generate.py::_load_workflow` 在「生圖當下」才回 422，使用者要等到生圖失敗才知道（即 5/19 的 AnimaStandardV5 事件）。

**修法**（`settings.py`）：把同樣的偵測前移到列表／切換階段。
- 新增 helper `_wf_is_ui_format(name)`（邏輯比照 `_wf_has_ipa`，偵測頂層 `nodes` 是否為 list）
- `GET /settings/workflows` 回傳新增 `invalid` 對照表，前端可在清單即標記不可用檔案
- `POST /settings/workflow` 切換到 UI 格式檔時，直接回 422 並附明確指引（用語與 `_load_workflow` 一致：請用 Save (API format) 重新匯出）

**風險**：低。純新增欄位與一處前置驗證，不影響既有 API format 工作流。

### 驗證狀態
- `ai_text.py`：`py_compile` 通過。
- `ai_art.py`、`settings.py`：因沙箱掛載同步延遲無法直接編譯，改以等價內容副本 `py_compile` + 執行確認語法與邏輯正確；檔案工具端（實際 `F:\wk\Craftflow`）為完整正確版本。

**風險**：低。唯讀輪詢，不影響既有狀態流。

### 優化 D：友善錯誤訊息映射（報告 3.2）

**根因**：元件直接顯示後端原始 detail（如 `[Ollama unavailable ... run \`ollama serve\`]`），對使用者不友善。

**修法**（`frontend/src/api/client.js`，集中於共用層）：
- 新增 `friendlyError(detail)`：僅針對「Ollama/ComfyUI 未啟動或連線失敗、逾時」等明確情境改寫為可行動中文；其餘（多為後端中文 detail，如 workflow UI 格式指引）原樣保留
- `request()` 在 `throw` 前套用 `friendlyError`；並補捉 fetch 網路層失敗 → 「無法連線到後端服務」

**涵蓋範圍**：所有走 `client.js`（`apiGet`/`apiPostJson`/`apiPostForm`…）的呼叫。**未涵蓋**：部分元件仍用原生 `fetch`（CharacterTab/GenerateTab 等），其錯誤處理各自獨立，列為後續；但狀態燈（優化 C）已能提前提醒服務離線，降低踩錯機率。

**風險**：低。僅改寫訊息字串，不改流程；未匹配的訊息原樣透傳。
