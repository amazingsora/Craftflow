# Craftflow 私人創作助理系統流程（System Flow v2）

## 1. 系統目標

**產品名稱：Craftflow**

本系統為**私人創作輔助工具**，協助創作者進行角色設計、世界觀構築與文本創作，
並提供情緒、節奏與一致性分析。

核心原則（v2 修訂）：

- **人類創作者永遠是決策主體** —— 任何 AI 產出皆為建議，從不覆寫原文。
- **AI 提供分析、提問與改寫建議** —— 改寫建議寫入獨立報告檔，原稿不動。
- **所有創作歷程皆可回溯（Git 時間軸）** —— 包含被採納與被否決的建議。

> v1 → v2 的差異：v1 將「改寫建議」視為違反設計哲學的行為。v2 釐清了
> **「修改原文」與「提供改寫文本作為參考」是兩件事**：原文絕對不被覆寫，
> 但分析模組可以產出獨立的 `*_rewrite.md` 給作者參考、比對、選擇性採納。

---

## 2. 系統總覽流程

整體流程自「創作輸入」開始，經過保存、分析、判定、回饋與**改寫建議**，
最終回到創作者本身，形成可重複的創作循環。

---

## 3. 流程分層說明

### 3.1 創作輸入層

創作者可輸入以下任一類型內容：

- 角色設定（人物背景、性格、動機）
- 世界觀資料（規則、勢力、歷史）
- 創作草稿（章節、片段、對話）

此層不進行任何分析或修正，僅作為創作事實的來源。

### 3.2 原始創作保存層（Git）

所有創作內容會以檔案形式保存，並納入 Git 版控系統。

設計重點：

- **原文不可被覆寫** —— 即使改寫建議被採納，採納動作也是建立新版本
- 每一次創作狀態皆可回溯
- 分析結果與改寫建議皆寫入獨立檔案，不修改原始檔案

### 3.3 分析觸發判定層

分析並非每次都自動執行。觸發條件至少其一：

- 創作者主動要求分析
- 單一文本超過指定字數門檻
- 系統偵測到潛在節奏或情緒異常

此層的目的是**避免過度打擾創作流程**。

### 3.4 創作分析模組（AI）

此模組為 AI 推論核心，負責多維度分析。包含三個子模組：

#### 3.4.1 情緒檢測（規劃中）

- 情緒張力變化曲線
- 情緒壓抑或爆發區段標記

#### 3.4.2 節奏檢測（v0.2 已實作）

- 段落字數異常（過短 / 過長）
- 連續長段（pacing drag）
- 後續擴展：句長變化、停頓密度、語速曲線

#### 3.4.3 一致性檢查（v0.3 新增）

- 角色行為是否符合既定設定
- 角色語氣是否符合 voice_style
- 世界觀規則是否被破壞
- 由 `worldbook/` 目錄下的 YAML 設定提供真實來源

採用兩階段設計：

1. **Surface scan**（無 LLM、必跑）—— 名稱／別名匹配 + 禁止關鍵字命中
2. **Semantic scan**（LLM、pro 模式啟用）—— 結構化 JSON 輸出隱性違規

分析結果為結構化 `ConsistencyIssue`，與原文並存於 `analysis/`。

### 3.5 分析結果輸出層

分析結果保存至獨立資料夾：

- `analysis/*_rhythm.md`
- `analysis/*_consistency.md`
- `analysis/*_emotion.md`（規劃中）
- `analysis/*_rewrite.md`（改寫建議報告）

特性：與原文並存、可被版控、可跨時間比較、**永遠不覆寫原文**。

### 3.6 創作助理介入判斷層

系統依 `intervention_rules` 與當前模式判斷如何回饋：

| 模式 | 節奏異常 | 一致性 high | 一致性 medium/low | LLM 用量 |
|------|----------|------------|--------------------|----------|
| gentle | 提出改寫建議 | 收進報告但不改寫 | 收進報告但不改寫 | 僅節奏改寫 |
| pro | 提出改寫建議（含連續長段） | 觸發 rewrite | 收進報告 | 改寫 + semantic scan |

模式設計目標：**尊重創作節奏與心理狀態**。

### 3.7 創作回饋呈現層

回饋以非侵入式方式呈現：

- 報告寫入獨立檔案，創作者選擇性閱讀
- 改寫建議與原文並列，**從不覆蓋原稿**
- 不要求即時回應

### 3.8 人類創作者決策層

最終決策永遠由創作者完成：

- 是否參考改寫建議
- 是否重構文本
- 是否更新 worldbook 設定（反向回饋）
- 是否繼續創作

決策結果回到流程起點。

## 4. 改寫引擎（v2 補述）

### 4.1 定位

**RewriteEngine 是建議產生器，不是文件編輯器。**

它讀入：段落原文 + 觸發原因（rhythm 異常 / consistency 違規）。
它輸出：獨立檔案 `analysis/*_rewrite.md` 中的建議段落。

它**從不**：

- 寫回原稿檔案
- 進入 Git commit 自動流程
- 在沒有 InterventionDecision 的情況下主動觸發

### 4.2 觸發來源

| 來源 | 觸發條件 | 模式 |
|------|----------|------|
| Rhythm | TOO_SHORT / TOO_LONG / 連續長段 | gentle / pro |
| Consistency | severity=high | pro only |

### 4.3 設計責任

- 改寫必須與原文同語言
- 改寫必須僅輸出段落本體（無前言、無解釋、無引號）
- 改寫不得新增劇情元素

---

## 5. Worldbook 結構（v0.3 新增）

```
tools/Craftflow/worldbook/
├── characters/<name>.yml      # 一個角色一檔
└── world/<name>.yml           # 世界觀，可多檔
```

角色 YAML 欄位：`name`, `aliases`, `core_traits`, `behavior_rules`,
`forbidden_actions`, `voice_style`, `notes`。

世界 YAML 欄位：`name`, `hard_rules`, `soft_rules`, `forbidden_keywords`,
`factions`, `notes`。

設定檔即真實來源；分析器**不擅自推論未定義內容**。

---

## 6. 硬體與模型的角色定位

GPU（目前為 RTX 5070 Ti）僅存在於：

- 「創作分析模組」中所有需要 LLM 的子流程
- 「改寫引擎」的推論階段
- 一致性檢查的 semantic scan 階段

系統設計保證：

- 更換顯卡不影響流程
- 增加顯卡僅提升效能
- 不產生專案結構耦合
- LLM 不可用時，所有非 LLM 路徑（rhythm 分析、surface scan）仍可運作

---

## 7. 結語

此系統的本質不是 AI 工具，而是一個**可長期陪伴創作的結構化思考環境**。

未來擴充（多模型、雙 GPU、即時介面、emotion 模組）皆應遵守本流程
所定義的責任邊界：**AI 提供分析與建議，原文絕不被覆寫，決策權永遠
屬於創作者**。
