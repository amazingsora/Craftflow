## 今日修改

### 1. ComfyUI Flux2 CLIPLoader 修正

**問題**：`canvas_expand_flux.json` 使用 `mistral_3_small_flux2_fp8.safetensors`，ComfyUI 載入時拋出 `ModuleNotFoundError: No module named 'transformers.models.pixtral'`。

**根因**：`transformers.integrations.mistral`（新 API，需 ≥4.47）不存在；fallback 的 `pixtral.convert_pixtral_weights_to_hf` 因 `_LazyModule` 在 ComfyUI 執行環境中被攔截而失效。

**修法**：在 `ComfyUI/comfy/text_encoders/flux.py` 的 `load_mistral_tokenizer()` 加第三層 fallback，直接 inline 定義 `MistralConverter`（只依賴 `tokenizers` + `bytes_to_unicode`，繞過 transformers import 問題）。

同時建立 shim：`transformers/integrations/mistral.py`（備用）。

### 2. Canvas Expand 策略重構（先生成再擴充）

**舊邏輯**：partial/bust coverage → 直接用 `_ipa_ref_bytes`（線稿/設計稿）做 Flux2 canvas expand，跳過 SDXL。

**問題**：線稿（line art）與 Flux2 全彩輸出風格不符，接縫明顯（上半身線稿 + 下半身全彩）。

**新邏輯**：
- partial/bust → **先跑 SDXL 生成完整風格圖**，再視情況 canvas expand
- Canvas expand 現只對 `coverage=full` 啟用（SDXL 有 CN 引導，輸出穩定為單人全身圖）
- partial/bust 的 SDXL 輸出不做 canvas expand（多視角設計稿套 expand 會出現飄浮問題）

### 3. Canvas Expand 參數調整

`canvas_expand_flux.json`：
- `steps`: 20 → 10（減少約一半時間）
- `grow_mask_by`: 8 → 48（大幅擴大邊界重算區，減少拼接線）

`art_generate.py`：
- `blend_px` 計算：`max(8, new_h//20)` → `max(20, new_h//8)`（Python 端梯度加寬）

### 4. 多視角設計稿問題（IPA + 線稿 reference）

**問題**：IPA reference 是多視角設計稿，IPA 0.1 仍會把多視角構圖帶入 SDXL 輸出。

**修法**：partial/bust 時在 positive prompt 加 `solo, single character`，防止設計稿格式。

### 5. CN bypass（保留）：線稿 Canny → 機甲效果

**實驗**：嘗試 partial/bust 改用低權重 CN（0.45）+ 早退（end_percent=0.40）。

**結果**：Canny 邊緣偵測在線稿輪廓線上，會被 SD 解讀為外骨骼/機甲關節邊緣，輸出品質反而變差。

**決策**：恢復 CN bypass。partial/bust → IPA only（線稿不適合做 Canny CN reference）。

## I — IPA / CN 權重策略確認（2026-06-07）

**使用者確認的策略**：
- **IPA**：前期草稿用 0.1（輕度風格染色，不強制外觀）；完稿調至 0.6（加強角色相似度）
- **CN**：0.5–0.9，甚至可到 1.0（AnimeLineArt 無機甲副作用，高權重安全）
- 設計原則：**CN 是外觀/形體主錨**，IPA 是輔助風格；兩者角色明確分開

**實施**：`Standard_V35.json` node 213 及 `_inject_ipa_cn_nodes` 動態注入已改為 `AnimeLineArtPreprocessor`（移除 `low_threshold`/`high_threshold` 參數，只保留 `resolution=1024`）。

## H — partial/bust 恢復 CN（取消 bypass，2026-06-07 傍晚）

**新限制條件（使用者確認）**：IPA 0.1~1（完稿調高）、CN(Canny/AnimeLineArt) **≥0.7 不可 bypass**、**否決 OpenPose**（後期動作無法統一）。

**根因再確認**：先前 G 的「機甲效果」來自 **Canny** preprocessor；目前 preprocessor 已是 **AnimeLineArtPreprocessor**（`_CN_UNION_TYPE` 仍含 canny 但實際走 anime_lineart），但程式仍沿用舊決策對 partial/bust 做 CN bypass（`_cn_ref_bytes = None`），違反新限制。

**變更（surgical）**：`art_generate.py` 兩處（char-gen ~1838、variant-gen ~2241）取消 partial/bust 的 CN bypass，改與 full 相同走 `_shrink_for_full_body`：partial 縮至畫布上半 58%、bust 42%，下半留空交給 SD 補腿，CN 以 AnimeLineArt 用使用者權重（≥0.7）引導上半身，單 pass 無接縫。`_COVERAGE_CN_WEIGHT`/`_COVERAGE_CN_END_PERCENT` 維持 None（沿用前端權重）。

**目標**：蔚藍檔案風乾淨全身商業人設圖（參考 ChatGPT 生成圖）。

**風險/待驗**：
- 縮圖後 CN hint 為上半比例，仍可能出現舊 A 的比例不符（生成角色只填上半 / 比例怪）；需實測確認 AnimeLineArt 是否優於 Canny。
- 下半身為 SD 腦補，腿部服裝/姿勢不保證符合設定（半身輸入物理必然）。
- 註：variant-gen 第 ~2249 行原「線稿 Canny 反效果」註解已隨本次移除。

**✅ 實測結果（2026-06-07）**：半身概念圖（IPA=0.10、CN=0.85、Standard_V35.json）生成**單一全身角色**，無設計稿、無接縫、無機甲失真 → **構圖問題解決，方案 H 採用**。耗時約 48s（視覺分析 5.8s + 翻譯 2.7s + 上傳 4s + ComfyUI 33.8s）。
- 證實「Canny → AnimeLineArt」是關鍵：舊 G 的機甲失真不再出現，CN 可在 partial/bust 全程 ≥0.7 引導而不反效果。
- 餘留：外觀相似度受 IPA=0.10 限制偏低（下半身服裝由 SD 自由發揮）。提升相似度走「完稿時調高 IPA」即可，不影響構圖。

## 三天失敗原因彙整（2026-06-05 ～ 06-07）

目標：半身/胸像概念圖（partial/bust）要生成**完整全身**且外觀貼合。基準：全身草圖（full）+ CN 高權重一向正常；問題只在半身輸入。

| 方案 | 做法 | 結果 | 失敗根因 |
|------|------|------|----------|
| A | CN + 草圖縮放（`_shrink_for_full_body`，fill≈0.58） | ❌ 三視圖設計稿 / CN 無效 | 縮圖 **Canny** 邊緣是小比例，生成角色全尺寸 → 比例不符無法引導；CN 退場後回設計稿 default |
| B | Two-pass（Pass1 生全身粗稿 → Pass2 當 CN ref） | ❌ Pass1 就出設計稿 | Pass1 無 CN + IPA 僅 0.10 → fabricatedXL 直接生設計稿，連環崩 |
| C | Inpainting（上半保留草圖、mask 下半補腿） | ⚠️ 有腳但接縫明顯 | 上半線稿風 + 下半 SD 全彩 → 風格不一致 |
| D | img2img（denoise=0.70 全圖渲染） | ❌ 角色差太多 | 70% 噪聲把草圖結構幾乎洗掉 |
| E | CN bypass + IPA only | ✅ 單一全身無接縫 / ⚠️ 不跟姿勢 | 完全靠 text+IPA；IPA=0.10 相似度低，且**違反新限制（CN 不可 bypass）** |
| F | Flux2 Canvas Expand（先生成再外擴下半） | ❌ 飄浮褲子 | bypass 下 SDXL 仍生多視角設計稿 → 外擴貼在設計稿底部 |
| G | CN 低權重早退（**Canny** 0.45/0.35 + early end） | ❌ 機甲/外骨骼失真 | **Canny 把線稿服裝輪廓誤判為機甲關節邊緣** |

**貫穿三天的兩條根因**：
1. **Canny 不適合線稿**：服裝輪廓線被誤讀成結構/機甲邊緣（A、G 的核心）。→ 解法＝改用 **AnimeLineArtPreprocessor**。
2. **線稿 vs 全彩的拼接**：任何「保留上半、另補下半」方案（C、F）都會接縫。→ 解法＝**單 pass 全圖一起生**（H：縮圖留白 + CN 引導上半 + SD 自然補下半）。

**最終結論（H）**：AnimeLineArt（取代 Canny）+ shrink 留白 + CN 全程 ≥0.7 + 全身 prompt tag → 一次生成乾淨全身角色，同時滿足三條限制（IPA 0.1~1、CN≥0.7、無 OpenPose）。被否決的方向：固定縮圖 Canny（A）、Two-pass（B，除非 Pass1 補 IPA+防設計稿）、Inpaint/Outpaint 補腿（C/F，接縫）、高 denoise img2img（D）、CN bypass（E，違限制）、Canny CN（G）。

## I — checkpoint 模式（AnythingXL_xl）構圖裁切：加高全身畫布

**問題**：生成模式切到 checkpoint（`text_to_image.json` + `AnythingXL_xl`）後，全身圖被裁切（小腿/腳被切），不像 Standard_V35 模式完整。

**根因分析（逐行確認，非漏套用全身邏輯）**：
- 全身路徑（coverage 偵測 → shrink/letterbox → 全身畫布 → full body 標籤 → CN 引導）在 checkpoint 模式**有正常套用**（`need_coverage` 成立、動態注入 IPA/CN 節點）。
- `EmptyLatentImage` 寬高覆寫（art_generate.py ~2097）對**所有**工作流生效；Standard_V35 內建雖為 1024×1536，實際也被覆寫成全身畫布值 → 兩模式跑相同畫布。
- 因此差異只剩 **checkpoint 模型本身**：`fabricatedXL_v70`（Standard 內建）在 832×1216 能框進全身；`AnythingXL_xl` 偏 portrait/半身構圖，同畫布下裁掉小腿/腳。

**變更（surgical）**：`art_generate.py` 全身畫布常數整體往上拉一個 SDXL 直幅 bucket（皆 64 倍數、約 1MP）：

| 級距 | 舊 | 新 |
|------|-----|-----|
| TALL（身高 ≥170） | 768×1344 | **704×1408** |
| STD（預設） | 832×1216 | **768×1344** |
| SHORT（<150/幼態） | 896×1152 | **832×1216** |

縱向空間加大後，portrait 偏向的 checkpoint 也能容下全身。

**風險**：full-body 友善的底模（如 fabricatedXL）在更高畫布下可能略多頭頂/腳下留白；屬可接受範圍。待實測各 checkpoint 效果。

**✅ 實測結果（2026-06-07）**：

| checkpoint / 工作流 | 結果 |
|------|------|
| `novaAnimeXL_ilV190` + checkpoint 模式（text_to_image.json） | ✅ 完整全身，可行（加高畫布奏效） |
| `fabricatedXL_v70` + Standard_V35.json | ✅ 完整全身，可行 |
| `AnythingXL_xl` + checkpoint 模式 | ❌ 下半身仍不生成 → **模型限制**（Anything 系列嚴重 portrait 偏向，加高畫布+CN 也逼不出全身），放棄逼全身 |

**機制澄清（重要）**：`Standard_V35.json` 在 `data/custom_workflows/`，屬自訂工作流；程式**刻意不對自訂工作流套用 checkpoint 選擇器**（`_load_workflow`：`found_in_custom` → 不覆寫 ckpt_name），Standard_V35 永遠用內建 `fabricatedXL_v70`。故「novaAnimeXL + Standard_V35」實際跑的是 fabricatedXL，**非** novaAnimeXL —— 使用者觀察到的「不正常」即源於此，非隨機。

**決策（2026-06-07）**：**維持現狀**，不改 checkpoint 覆寫設計。使用建議：全身人設用 `novaAnimeXL`（checkpoint 模式）或 `fabricatedXL`（Standard_V35）；`AnythingXL` 不用於全身。
