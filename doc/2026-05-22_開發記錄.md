## 1. 歷史記錄 Bug 修復 — 角色人設圖未加入列表

### 問題
角色管理頁生成人設圖後，歷史記錄列表沒有新增項目。其他 Tab（GenerateTab、ProcessTab、ComposeTab）都正常。

### 根本原因
`App.jsx` 傳 `onAddHistory` prop 給三個 Tab，但 `CharacterTab` 的 JSX 沒有傳：
```jsx
<CharacterTab />  ← 漏傳 onAddHistory
```

### 修復
- `App.jsx:504`：`<CharacterTab onAddHistory={addHistory} />`
- `CharacterTab.jsx`：主元件接收 `{ onAddHistory }`，往下傳給 `CharacterDetailView`
- `CharacterDetailView`：接收後在 `generateDesignImage`（主版本）和 `generateVariantDesignImage`（Tab 2/3）生成成功後呼叫：
  ```js
  onAddHistory?.({ type: 'character', url, filename: `${char.name}_design_${Date.now()}.png`, label: `${char.name} 人設圖` })
  ```

## 2. Ollama 錯誤字串混入 Prompt Bug 修復

### 問題
角色人設圖生成時，最終 prompt 出現：
```
..., [Ollama error at http://localhost:11434: 404 Client Error: Not Found ...], character design sheet, ...
```

### 根本原因
`compiler.py` 呼叫 Ollama 失敗時，`ollama_client.generate()` 回傳 error string（以 `[` 開頭），`_extract_output` → `_sanitize_to_list` 把整串錯誤當成一個 SD tag，最後混進 positive prompt。

實際根源：`.env` 設 `TEXT_MODEL=dolphin-llama3`，但該模型未安裝，Ollama 回傳 404。

### 修復
- `compiler.py`：`raw_response.startswith("[")` → `raise RuntimeError(raw_response)`
- `art_generate.py`：三處 `compile_prompt` 呼叫加 `try/except RuntimeError` → `HTTPException(503)`，錯誤訊息包含模型名稱提示安裝

## 4. Prompt 管理系統 — 外部化為實體文字檔

### 需求
所有 AI Prompt 散落在 Python service 檔案中（hardcode 字串），難以維護與快速調整。
希望：
1. 每個 prompt 獨立存為 `.txt` 文字檔，可直接用編輯器修改
2. 找不到對應檔案時，程式自動使用 hardcoded 預設值，不影響正常運作

### 盤點結果
共 **14 個 prompt** 分散在 4 個 service 檔：

| Service | 數量 | Prompt |
|---|---|---|
| `art_service.py` | 6 | 草稿分析、完稿評審、配色建議、構圖分析、插畫描述、創作顧問 |
| `character_service.py` | 4 | 角色摘要整理、從章節提取特徵、設計問答、插圖視覺描述 |
| `consistency_service.py` | 1 | 一致性檢查（含 JSON schema） |
| `rewrite_service.py` | 3 | 改寫主模板、gentle 指令、aggressive 指令 |

### 實作

**新增 `backend/app/services/ai/prompt_loader.py`**
- `load_prompt(key, **kwargs)` — 讀取 `backend/app/prompts/{key}.txt`，找不到用 `_DEFAULTS[key]` 回退
- `kwargs` 非空時套用 `str.format_map()` 替換變數（`{var_name}`）
- 含 JSON 格式範例的 prompt，literal 大括號需用 `{{` / `}}` 轉義（和 Python f-string 規則一致）

**新增 `backend/app/prompts/` 目錄**
```
backend/app/prompts/
├── art/
│   ├── sketch_critique.txt
│   ├── finished_critique.txt
│   ├── line_color.txt
│   ├── composition_analysis.txt   ← 含 {user_question}
│   ├── describe_illustration.txt
│   └── composition_advisor.txt    ← 含 {question}
├── character/
│   ├── generate_summary.txt       ← 含 {name}, {raw_notes}
│   ├── extract_from_text.txt      ← 含 {character_name}, {chapter_text}
│   ├── design_chat.txt            ← 含 {character_name}, {profile_ctx}, {question}
│   └── describe_portrait.txt      ← 含 {character_name}
├── consistency/
│   └── check.txt                  ← 含 {char_desc}, {paragraph}
└── rewrite/
    ├── rewrite.txt                ← 含 {instruction}, {reason}, {paragraph_text}
    ├── instruction_gentle.txt
    └── instruction_aggressive.txt
```

**修改 4 個 service**：移除 hardcoded 常數與 f-string prompt，全改呼叫 `load_prompt(key, **kwargs)`

### 使用方式
- **修改 prompt**：直接編輯對應 `.txt` 檔，無需動 Python 程式碼，重啟後生效
- **刪除 `.txt` 檔**：程式自動回退至 `_DEFAULTS`，行為與改前完全一致
- **變數替換規則**：`{variable_name}` 為佔位符；想在 prompt 裡寫 literal `{` `}` 時用 `{{` `}}`

## 5. 異色瞳（Heterochromia）Prompt 編譯支援

### 問題
角色特徵設有「(左眼為紅色，右眼為綠色的異色瞳)」時，生成的 SD prompt 完全缺少 `heterochromia` 及個別眼色 tag。

### 根本原因（雙重）
1. **Anchor regex 盲區**：`lexicon.py` 的 `EYE_RE` 只能匹配 `X色眼睛` 格式，對 `左眼為X色` / `右眼為Y色` 的描述方式完全抓不到，anchor 系統不介入
2. **LLM few-shot 缺失**：`_SDXL_TEMPLATE` 沒有異色瞳的輸入/輸出範例，換新模型後翻譯更容易失敗

### 修復

**`compiler.py`** — 新增 `_inject_heterochromia()`：
- 偵測到 `異色瞳` / `異色眼` 就強制把 `heterochromia`、`{color} eye (left)`、`{color} eye (right)` 插入 tag 列表頭部
- 同時移除 LLM 可能誤輸出的單色 `X eyes` tag（避免衝突）
- 在 `_clean_clothing_hallucinations()` 與 anchor extraction 之間執行，完全不依賴 LLM 能否正確翻譯

**`styles.py`** — 雙重防護：
- `_DANBOORU_COMMON_RULES` 加 `HETEROCHROMIA` 規則，讓 LLM 知道格式
- `_SDXL_TEMPLATE` 加對應 few-shot example

### ⚠️ KeyError 插曲
`_DANBOORU_COMMON_RULES` 說明文字中使用了 `{{left-color}}`，在 f-string 展開後變成 `{left-color}` 留在 template 中，`.format(prompt=text)` 呼叫時試圖替換 → `KeyError: 'left-color'`。

修復：把說明文字改成純文字描述，不使用任何 `{...}` 占位格式。

## 6. AI 整理摘要無法產生 — Qwen3 Thinking Mode

### 問題
將文字模型切換為 `fredrezones55/Qwen3.5-Uncensored-HauhauCS-Aggressive:9b` 後，角色「AI 重新整理」回傳空回應，前端無法顯示摘要。

### 根本原因
Qwen3 系列模型預設開啟 **thinking mode**（思考模式），Ollama 的 `/api/generate` 在此模式下：
- 思考過程放在 `<think>...</think>` 內部 token
- `response` 欄位**僅包含可見輸出**（thinking 區塊以外的文字）
- 若模型把所有 token 花在思考上、沒有輸出可見文字 → `response = ""`

`ollama_client.generate()` 執行 `r.json().get("response", "").strip()` 得到空字串，`summarize_character()` 存入空 `ai_summary`，前端顯示佔位文字，使用者誤認為「無法產生」。

### 修復

**`ollama_client.py`** — `generate()` payload 加 `"think": False`：
```python
payload = {"model": model, "prompt": prompt, "stream": False, "think": False}
```
- Qwen3 系列：停用 thinking mode，直接輸出可見回應
- 其他模型（dolphin-llama3 等）：忽略 `think` 欄位，行為不變

**`characters.py`** — `summarize_character()` / `summarize_variant()` 加錯誤偵測：
```python
if not summary or summary.startswith("["):
    raise HTTPException(503, detail=f"Ollama 回傳錯誤（模型：{used_model}）：{summary or '空回應'}")
```
Ollama 錯誤不再靜默存入 DB，前端可以顯示明確錯誤訊息。

### 歷史記錄（App.jsx）
- `formatTime` 改為 `月/日 HH:MM` 顯示
- 加 `_shortModel()` helper（strip 副檔名、截斷到 14 字）

- 縮圖 meta 第三行顯示 model（截斷，hover 顯示全名）
- Lightbox 頁腳也顯示 model

### AI 提示詞翻譯 bug 修正（art_generate.py）

**根本原因（質量標籤重複）**：`compile_prompt` 呼叫 ai_prompt 時未傳 `quality_prefix_override=""`，導致 `extra_compiled` 已包含 `masterpiece, best quality` 前綴，最終 prompt 出現 quality prefix 兩次。

**修正**：
```python
_ai_overrides = {**_overrides, "quality_prefix_override": ""}
extra_compiled, _ = compile_prompt(..., **_ai_overrides)
```

**可觀察性（CharacterTab.jsx + art_generate.py）**：
- 後端加 `X-AI-Prompt-Compiled` response header（base64，與 `X-Prompt` 同格式）
- 失敗時回傳 `"[compilation_failed]"` 字串
- 前端 DEBUG PROMPT 區塊新增「AI 提示詞編譯結果」，成功顯示黃色，失敗顯示紅色警告
