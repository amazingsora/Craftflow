## 二、後端：共用與去重

### 2.1 兩大設計圖函式去重 ⭐ 最高價值

**問題**
`generate_character_design`（1329–1655）與 `generate_variant_design`（1657–1941）兩個 async 端點，逐行 diff 後 285 行中約 214 行相同（~75%）。兩者簽章除 `slot: int` 外完全一致，且呼叫同一組 helper（`_resolve_style`、`_compile_overrides`、`_inject_ipa_cn_nodes`、`_detect_body_coverage`、`_shrink_for_full_body`、`_run` …）。

這也是當前最活躍的開發區（見 `2026-06-02_開發記錄.md`），每次調整都要**改兩個地方**，極易漏改、產生行為不一致。

**方案**
抽出共用核心，兩端點退化為「取資料 + 呼叫 pipeline」的薄殼：

```python
# services/ai/character_design.py（新檔）
async def run_design_pipeline(
    *, character: Character, target: DesignTarget,   # target 封裝「主圖」或「變體 slot」差異
    expression, art_style, use_ai_prompt, use_outfit,
    use_ipa, ipa_weight, use_controlnet, cn_weight, db,
) -> bytes:
    # 1. 解析畫風 / 組 prompt / outfit
    # 2. 載入 workflow → 注入 LoRA / IPA / ControlNet
    # 3. 全身模式偵測 body coverage → 縮放 CN 參考
    # 4. await guardian.request_focus(...) → _run(wf)
    ...
```

```python
# api/art_generate.py（薄殼）
@router.post("/characters/{cid}/generate-design")
async def generate_character_design(cid: int, ...):
    char = _get_character(db, cid)
    return Response(await run_design_pipeline(character=char, target=MainTarget(), ...),
                    media_type="image/png")

@router.post("/characters/{cid}/variants/{slot}/generate-design")
async def generate_variant_design(cid: int, slot: int, ...):
    char = _get_character(db, cid)
    return Response(await run_design_pipeline(character=char, target=VariantTarget(slot), ...),
                    media_type="image/png")
```

差異點（主圖 vs 變體：portrait 來源、存檔路徑、variant 欄位）以小的 `DesignTarget` 物件或策略函式封裝，而非整段複製。

**風險**：中。屬活躍開發核心，需保證行為完全等價。
**緩解**：先寫「黃金樣本」測試 —— 用固定 seed 對同一角色跑改動前 / 後，比對輸出 bytes 或 workflow JSON 一致，再合併。
**預估**：0.5–1 天。

### 2.2 `ollama_client` 錯誤處理樣板去重

**問題**
`generate` / `analyze_image` / `analyze_image_bytes` / `analyze_multi_images_bytes` 四函式，每個都重複：組 payload → `requests.post(.../api/generate)` → `raise_for_status` → 兩段一模一樣的 `except ConnectionError` / `except Exception` 回傳 `[Ollama/Vision unavailable/error ...]` 字串。

**方案**
抽單一私有送出函式，四個公開函式只負責組 payload：

```python
def _post_generate(payload: dict, timeout: int, label: str) -> str:
    try:
        r = requests.post(f"{OLLAMA_BASE}/api/generate", json=payload, timeout=timeout)
        r.raise_for_status()
        return r.json().get("response", "").strip()
    except requests.exceptions.ConnectionError:
        return f"[{label} unavailable at {OLLAMA_BASE}: run `ollama serve` ...]"
    except Exception as e:
        return f"[{label} error at {OLLAMA_BASE}: {e}]"
```

**風險**：低。純內部重構，公開簽章與回傳字串格式不變。
**預估**：1–2 小時。

### 2.3 `comfyui_client` 輪詢去重

**問題**：`wait_for_result`（取 images）與 `wait_for_text_result`（取 text/tags）輪詢骨架相同，只差「從 outputs 取什麼」。

**方案**：抽 `_poll_history(prompt_id, timeout, extract)`，`extract` 為「從單一 node output 取值」的 callable。

**風險**：低。
**預估**：1 小時。

### 2.4 `guardian.request_focus` 樣板收斂

**問題**：`await guardian.request_focus("ollama" | "comfyui")` 在端點中複製約 26 次。

**方案（擇一，皆低侵入）**
- **A**：包進 service 入口 —— 例如在 `_run()` 內部先確保 `comfyui` focus、在 `ollama_client` 送出前確保 `ollama` focus，端點不再手動呼叫。（注意 `_run` 為同步、`request_focus` 為 async，需配合 §5.1 一起處理）
- **B**：FastAPI dependency，如 `_=Depends(focus_comfyui)`，宣告式取代手動 await。

**風險**：中（改變「何時切換 VRAM」的時機，需驗證不會在同一請求內來回切換造成反覆載入 / 卸載模型）。
**預估**：半天，建議排在 §5.1 之後。

## 三、前端：統一 API 層

### 3.1 共用 `apiClient` ⭐ 最高價值

**問題**
無任何共用層。49 處 `fetch` 散落，base URL 三種寫法（§1）。`CharacterTab.jsx` 自己寫了很好的 `apiFetch` / `apiDelete`（200–224 行），但**只在該檔內使用**，其餘 7 個元件各自 raw fetch + 各自 `res.ok` 判斷 + 各自 `.json().catch(...)`。新增端點要在多處重貼相同樣板。

**方案**
新增 `src/api/client.js`，把 `CharacterTab` 既有的 `apiFetch` 模式升級為全專案共用：

```js
// src/api/client.js
const BASE = '/api/v1'   // 單一來源

async function request(path, opts = {}) {
  const r = await fetch(`${BASE}${path}`, opts)
  if (!r.ok) {
    const err = await r.json().catch(() => ({ detail: r.statusText }))
    throw new Error(err.detail || r.statusText)
  }
  return r            // 由呼叫端決定 .json() / .blob()
}

export const apiGet      = (p)        => request(p).then(r => r.json())
export const apiPostJson = (p, body)  => request(p, { method:'POST', headers:{'Content-Type':'application/json'}, body:JSON.stringify(body) }).then(r => r.json())
export const apiPostForm = (p, fd)    => request(p, { method:'POST', body: fd })   // FormData，不設 Content-Type
export const apiPostBlob = (p, body)  => request(p, { method:'POST', ...body }).then(r => r.blob())
export const apiDelete   = (p)        => request(p, { method:'DELETE' })
```

集中端點路徑（可選但建議）：

```js
// src/api/endpoints.js
export const EP = {
  compilePrompt: '/art/compile-prompt',
  generate:      '/art/generate',
  ipadapter:     '/art/ipadapter',
  imgGuide:      '/art/img-guide',
  lineart:       '/art/lineart',
  // ...集中管理，端點改名只改一處
}
```

**遷移策略**：逐檔替換，一次一個元件，每換完一個就手測該頁。`CharacterTab` 因已有同形 `apiFetch`，改為 `import` 共用版即可，風險最低，先做它當示範。

**風險**：低–中（逐檔遷移，影響面可控）。
**預估**：每個元件 0.5–1 小時，全部約 1 天。

### 3.2 `useAsync` hook 統一 loading / error 樣板

**問題**
每個元件重複宣告 `const [loading,setLoading]=useState(false)` / `const [error,setError]=useState(null)`，再到處 `setLoading(true)…try…catch(setError)…finally(setLoading(false))`（如 `GenerateTab.jsx` 137/143/226/289/292）。

**方案**

```js
// src/api/useAsync.js
export function useAsync() {
  const [loading, setLoading] = useState(false)
  const [error, setError] = useState(null)
  const run = useCallback(async (fn) => {
    setLoading(true); setError(null)
    try { return await fn() }
    catch (e) { setError(e.message); throw e }
    finally { setLoading(false) }
  }, [])
  return { run, loading, error, setError }
}
```

呼叫端：`const { run, loading, error } = useAsync()` → `await run(() => apiPostForm(EP.generate, fd))`。

**風險**：低。
**預估**：hook 0.5 小時 + 各元件接入。

## 五、效能與穩定性

### 5.1 同步阻塞卡死 event loop ⭐⭐ 最嚴重

**問題（根因）**
`_run(workflow)`（282 行）是**同步**函式，內部 `comfyui_client.wait_for_result` 同步輪詢，最長 `timeout=300s`。它被 7 個 **`async def`** 端點直接呼叫（357、411、1123、1632、1916、2073、2139）。

在 FastAPI 中，`async def` 端點裡的同步阻塞會**卡住整個 event loop**：生一張圖的數十秒內，後端無法處理任何其他請求 —— 包含前端輪詢進度、`/status` 健康檢查。這直接抵觸 CLAUDE.md 的「**進度回饋**」要求：event loop 被佔住時，前端根本拉不到進度。Ollama 的同步 `requests`（vision 分析 `timeout=300`）同理。

**方案（最小變更，不改邏輯）**
用 Starlette 內建 threadpool 把阻塞呼叫移出 event loop：

```python
from starlette.concurrency import run_in_threadpool
# 端點內：
image_bytes = await run_in_threadpool(_run, wf)
```

Ollama 端可同樣包裝，或將 `ollama_client` / `comfyui_client` 改用 `httpx.AsyncClient`（較大改動，列為後續）。

**風險**：低–中。`run_in_threadpool` 是官方建議做法；需注意 thread 內不可再 await，且 DB session 若跨 thread 使用要留意（本案 `_run` 不碰 DB，安全）。
**預估**：半天。**建議列為第一優先施工項**（價值最高、風險低）。

### 5.2 ComfyUI 輪詢間隔

**問題**：`wait_for_result` / `wait_for_text_result` 固定 `time.sleep(2)`。短任務最多被多等近 2s，長任務則輪詢過密。

**方案**：漸進 backoff（如 0.5s 起，封頂 3s）。屬微優化，與 §5.1 同檔處理順手即可。
**風險**：低。

### 5.3 前端 localStorage history 配額

**現況**：`App.jsx` 已有 try/catch 降級（滿了縮到 40 筆）—— 處理得當。
**建議**：上限明確化（`_MAX_HISTORY=100`）；若日後縮圖增多，評估改 IndexedDB。非急迫。

### 5.4 安全：CORS

**現況**：`main.py` `allow_origins=["*"]`，已註記 Phase 4 收斂。本機階段可接受，上線前改白名單。

### 5.5（記錄）`services/concurrency.py` 已移除

grep 發現殘留 `concurrency.cpython-*.pyc`，但 `.py` 已不存在 —— 表示曾有並發控制層後被移除。若 §5.1 後仍出現「多請求同時打 ComfyUI」的競爭，再評估是否需要重新引入一個輕量序列化鎖（例如同時只允許一個 `_run`）。**先不主動加**，避免過度設計。

## 八、施工守則（沿用專案規則）

1. 每項改動前讀相關檔案，先確認再動手。
2. Surgical Update：一次一個模組 / 一個元件，獨立 commit，可單獨回退。
3. 去重 / 搬家類改動要求**行為等價**：以固定 seed 的黃金樣本（輸出 bytes 或 workflow JSON）比對前後一致。
4. 拆檔每步都重啟後端確認 `/docs` 載入、前端手測對應頁。
5. 技術細節與當日進度寫入 `doc/YYYY-MM-DD_開發記錄.md`，本規劃僅更新狀態與 checkbox。
