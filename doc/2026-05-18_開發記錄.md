## 3. 角色管理多版本 Tab

## 4. 當日除錯與修補

### 4-1. Python 3.9 啟動錯誤修正

**問題**
Docker 環境為 Python 3.9，`app/api/characters.py` 的 `VariantUpdate(BaseModel)` 使用 `str | None` 語法，Pydantic v2 自行評估 annotation 時拋出：
```
TypeError: Unable to evaluate type annotation 'str | None'
```

**修正**
- `backend/app/api/characters.py` 頂部加入 `from __future__ import annotations`
- 將 `VariantUpdate` 所有 `str | None` / `int | None` 改為 `Optional[str]` / `Optional[int]`（Pydantic v2 的 `eval_type_backport` 不支援 `|` 語法，需明確使用 `Optional`）

---

### 4-2. AI 人設圖背景污染視覺提取修正

**問題**
1. **代表色背景滲入髮色**：Vision model（qwen2.5vl）分析概念圖時，將圖片背景色（由角色代表色 `character.color` 轉換的 SD 背景 tag）誤判為角色髮色。
2. **背景 tag 重複出現**：`raw_desc` 裡混入英文 `bg_color_name`，由 Ollama 編譯後輸出一次；`bg_tag` suffix 又追加一次，SD 收到重複背景指令。

**修正**（`backend/app/api/art_generate.py`）

| 項目 | 修改內容 |
|---|---|
| `_visual_extract_prompt` | 將「忽略背景顏色」指令移至 prompt **最前面**，並加重標示；在髮色項目補充「不受背景影響」 |
| `generate_character_design` | 從 `raw_desc parts` 移除背景描述，只保留 suffix 的 `bg_tag`（單次出現，不過 Ollama） |
| `generate_variant_design` | 同步移除 `raw_desc` 裡的背景 part |

### 4-3. Tab 2 概念圖上傳修正（前端）

Tab 2 / Tab 3 的「+」按鈕使用 `vConceptRefs[activeTab - 1].current.click()` 觸發隱藏 file input，ref 在某些情況下尚未附加，導致上傳無法啟動。

### 4-4. Variant 概念圖 / AI 圖存檔失效（SQLAlchemy JSON mutation）

**問題**
Tab 2 / Tab 3 上傳概念圖或儲存 AI 人設圖時，後端回傳 200 OK，但前端取得的 `concept_images` / `ai_generated_images` 仍為空陣列 `[]`，畫面無更新。

**根本原因**
`_get_variants(character)` 做的是 **shallow copy**（`list(character.variants or [])`），因此 `variants[idx]` 與 `character.variants[idx]` 指向**同一個 dict 物件**。
當我們修改 `variants[idx]["concept_images"] = [...]` 後，`character.variants[idx]` 也同步被修改。
SQLAlchemy 在 flush 時比較新舊值，發現兩者相同（指向相同 dict），判定為「無變更」，**跳過 UPDATE**，導致資料庫從未寫入。

**確認過程**
前端 log（`[vConc] serverVar.concept_images: []`）與後端 200 OK 並存，明確指向後端回應值有誤，縮小至 DB 寫入層問題。

**修正**（`backend/app/api/characters.py`）

```python
from sqlalchemy.orm.attributes import flag_modified
```

在所有 6 個 variant 寫入路由的 `character.variants = variants` 之後、`db.commit()` 之前，統一加入：

```python
flag_modified(character, 'variants')
```

受影響端點：
- `PUT /characters/{id}/variants/{slot}` — 更新文字欄位
- `POST /characters/{id}/variants/{slot}/concept-images` — 上傳概念圖
- `DELETE /characters/{id}/variants/{slot}/concept-images/{idx}` — 刪除概念圖
- `POST /characters/{id}/variants/{slot}/ai-images` — 儲存 AI 圖
- `DELETE /characters/{id}/variants/{slot}/ai-images/{idx}` — 刪除 AI 圖
- `POST /characters/{id}/variants/{slot}/summarize` — AI 整理

**附帶改善**（`frontend/src/components/CharacterTab.jsx`）
- `saveVariantPendingFirst`：將 `pendingQueue: v.pendingQueue.slice(1)`（stale closure）改為 `setVs(prev => prev.map(...))` functional update，與 Tab 1 的 `setPendingQueue(prev => prev.slice(1))` 模式一致。
- 「捨棄」按鈕新增 `disabled={vState.savingGen}`，防止儲存進行中同時 revoke blob URL。

## 5. 提示詞引擎重構與視覺權重優化 (Gemini CLI)

### 背景
為了提升提示詞生成的穩定性與可維護性，並解決視覺分析（Vision）在單色背景下產生的「幻覺」與文字描述衝突的問題，由 Gemini CLI 進行了核心提示詞引擎的深度重構。

### 重構與優化內容

#### A. 結構化與型別安全 (`styles.py`)
- **Pydantic 模型化**：引入 `StyleConfig` BaseModel，將原本的字典配置轉為具備型別檢查的物件。
- **自動標籤過濾同步**：新增 `model_validator`，現在定義 `quality_prefix` 時，系統會自動將其拆分並注入 `banned_tags`，徹底消除品質標籤重複堆疊導致權重崩壞的 Bug。

#### B. 模板模組化與 DRY 原則 (`styles.py`)
- **共通規則提取**：將顏色翻譯表、Danbooru 格式要求、禁令規則等提取至 `_DANBOORU_COMMON_RULES`，實現跨模型風格的配置同步。
- **動態模板組合**：利用 Python 格式化字串組合 LLM Template，未來修改翻譯邏輯時只需單點更新。

#### C. 視覺與文字衝突解決方案 (`compiler.py` & `art_generate.py`)
- **確立優先權規則 (Conflict Arbitration)**：在 LLM 模板中明確定義 `CONFLICT` 規則。當「視覺特徵」與使用者手寫的「優先採用（外貌與個性）」衝突時，後者具備絕對優先權。
- **強化髮型長度錨點 (Trait Anchoring)**：優化了顏色錨點邏輯，現在會額外鎖定「頭髮長度」（如：短髮、長髮）。即使 LLM 因為視覺分析誤判長度，編譯器也會強制修正並清除衝突標籤。
- **視覺分析「背景防禦」**：在視覺提取提示詞中加入強烈的警告指令，要求 AI 忽略單色背景（如全粉色背景），只觀察角色輪廓內的特徵，降低背景色污染提示詞的機率。

### 驗證結果
- **Pony / SDXL**：均能正確自動過濾品質標籤。
- **衝突測試**：當視覺分析判定為「長粉色頭髮」但文字描述為「短褐色頭髮」時，最終產出的提示詞能正確修正為 `brown hair, short hair, short twin tails`。
