## 痛點 1：前端拆分 IPA / ControlNet 控制 ✅

### 根因
`CharacterTab.jsx` 把 `use_controlnet` 綁在 `ipaEnabled` 同一個 checkbox，`cn_weight` 寫死 `0.7`，導致「一個開關控制兩者、slider 只動 IPA」。

### 改動（`frontend/src/components/CharacterTab.jsx`）
- 主角色新增 state：`cnEnabled`（預設 true）、`cnWeight`（預設 0.7），比照 `ipaEnabled`/`ipaWeight` 走 `_loadGenPrefs`/`_saveGenPref` 持久化。
- 變體：`_initVariant` 預設與回傳補 `cnEnabled`/`cnWeight`；UI 與送出讀 `vState.cnEnabled ?? true`、`vState.cnWeight ?? 0.7`（向後相容舊偏好）。
- UI：在 IPA「概念圖參考」旁新增獨立「ControlNet」checkbox，並新增「CN 強度」slider（0.1–1.5）。主角色與變體兩處皆處理。
- 送出參數：`use_controlnet` 改用 `cnEnabled`、`cn_weight` 改用 `String(cnWeight)`。

### 風險
低。後端早已分離 `ipa_weight`/`cn_weight` 兩參數，本次純前端。舊偏好缺 `cnEnabled`/`cnWeight` 時以預設值補足，不會讀成 undefined。

## 痛點 2：ControlNet 可單獨停用（bypass）✅

### 方案
採「切換即生效」的 bypass 路線（單一 workflow + UI 開關），符合「設置簡單、生圖快速」需求：CN 關閉時直接剝離節點，少跑 preprocessor / 不載入 controlnet 模型。

### 改動（`backend/app/api/art_generate.py`）
- 新增 `_bypass_controlnet_nodes(wf)`：
  - `ControlNetApplyAdvanced` 為雙輸出（positive 槽 0 / negative 槽 1），把消費端還原接回 apply 節點自身的上游 `positive`/`negative` conditioning。
  - 由 `control_net`、`image` 邊向上游 BFS 收集 CN 專屬節點（apply、ControlNetLoader、SetUnionControlNetType、preprocessor、CN LoadImage）後移除。
  - 安全檢查：仍被 CN 鏈以外節點引用的節點不刪，避免誤刪共用上游。
- 主角色與變體流程：當 `_cn_ref_bytes is None`（CN 停用或無參考圖）且工作流含 CN 時呼叫 bypass。位置在 `_inject_prompts` 之前，使 KSampler 條件線改為直連 `CLIPTextEncode`（Pass 1 直注入，比原本 fallback 更可靠）。

### 驗證
以 `Standard_V35.json` 實測：KSampler `positive`/`negative` 由 `['214',0]/['214',1]` 正確重接為 `['43',0]/['44',0]`；CN 五節點（210/211/212/213/214）全部移除；conditioning 節點 43/44 保留；無 dangling 參考；節點數 46 → 41。完整檔案 `py_compile` 通過。

### 風險
重接 conditioning 線若錯會導致 prompt 不生效或 ComfyUI 422，已用實測驗證重接正確。

## 追加：全身畫布比例「自動匹配大小」✅

### 需求
使用者測試後：全身效果可接受，但希望畫布能「自動匹配大小」（角色完整入框 + 比例依身形）。明確要求頭頂裁切的 CN 權重 / OpenPose 機制**先不動**。

### 改動（`backend/app/api/art_generate.py`，主角色 + 變體）
- 新增 `_fullbody_canvas(height)`：依 `character.height`（cm）自動選全身畫布比例。
  - `≥170` → `768×1344`（高瘦/長腿，更長縱向）
  - `150–169` → `832×1216`（標準，預設）
  - `<150` → `896×1152`（矮/幼態，較方）
  - 身高未知或無法解析 → 標準。
- 三組比例皆 64 倍數、約 1MP，貼近 SDXL 訓練分佈。
- 全身分支由固定 `832×1216` 改為 `width, height = _fullbody_canvas(...)`。
- 更長的縱向畫布順帶給頭/腳更多空間，**間接減少頭頂裁切**（不碰 CN）。

### 驗證
`_fullbody_canvas` 邊界值（None/''/'abc'/140/149/150/160/170/175/180）邏輯全對；三組尺寸 MP 與 64 倍數檢查通過；完整檔案 `py_compile` 通過。

### 未解（使用者選擇先不動）
頭頂裁切根因仍是 CN Canny 高權重複製概念圖構圖。徹底解決需「全身模式降 CN 權重」或「OpenPose 標準骨架」，待使用者要求再做。

## 追加：CN 預設 0.85 + 全身模式接 OpenPose 標準骨架 ✅

### CN 權重預設 0.7 → 0.85
使用者實測：CN 需 ≥0.85 才夠貼合概念圖。前端 `CharacterTab.jsx`（主角色 state、`_initVariant`、變體送出與 UI fallback）與後端 `art_generate.py` 兩個 endpoint 簽章 `cn_weight` 預設皆改 0.85。舊角色 localStorage 既有設定不受影響。

### 全身模式 OpenPose（解決「高 CN 又裁頭」）
高 Canny 權重會複製概念圖構圖（含裁頭）。改為注入**預渲染標準全身骨架**，使高 CN 鎖定的是標準全身姿勢而非概念圖框架 → 頭到腳完整。
- **骨架資產**：`data/pose_refs/fullbody_front.png`（832×1216，OpenPose 18-keypoint 標準配色，正面站姿、置中、上下留白）。以 PIL 程式生成，非 DWPreprocessor 即時運算 → 不需 onnx、不需使用者中段原圖（那會抽出殘缺骨架）、無節點 schema 風險。
- **新增 `_apply_openpose_fullbody(wf, skeleton_name)`**：① `SetUnionControlNetType.type → openpose`；② `ControlNetApplyAdvanced.image` 直接接 CN `LoadImage`（繞過 Canny preprocessor）；③ 移除 Canny 節點；④ `LoadImage` 載入骨架圖。
- **流程接入**（主角色 + 變體）：全身模式（`not is_expression`）且骨架檔存在時走 OpenPose；**fallback**：表情模式或套用失敗 → 沿用 Canny + 概念圖。概念圖外觀仍由 IPA 傳遞（外觀靠 IPA、姿勢靠骨架，職責分離）。
- **新增 `X-CN-Mode` header**（`none`/`canny`/`openpose`）方便前端與除錯確認實際走哪條。

### 驗證
- `_apply_openpose_fullbody` 實測 `Standard_V35.json`：union type=openpose、`214.image=['210',0]`、Canny 節點移除、骨架注入 LoadImage、IPA 鏈完好、無 dangling。
- 骨架圖視覺確認為正面站姿、頭到腳完整、置中留白。
- 後端最大可編譯前綴含全部改動，`py_compile` 通過。

### 取捨提醒
OpenPose 高權重會把姿勢鎖成標準站姿（人設圖通常可接受）；外觀一致性改由 IPA 承擔，IPA 太低（如 0.15）可能失真，建議全身 OpenPose 時適度提高 IPA。

## 修正：撤回 OpenPose，改以「概念圖補邊（letterbox）」解決中段裁切 ✅

### 使用者回饋
- IPA >0.4 即走樣 → 外觀不能靠 IPA，需低 IPA。
- OpenPose 標準骨架不行 → **姿勢必須跟隨上傳草圖**（固定站姿把姿勢蓋掉）。
- 要回到「Canny 跟隨草圖」的上一版，但解決「只生成中段」。

### 根因（重新定位）
檢視 char=1 概念圖 `concept_1_*.png`：**草圖本身是完整全身**（頭到腳），並非缺頭缺腳。
真正原因是**比例不符**：草圖窄長（303×691，AR≈0.44），生成畫布較寬（832×1216，AR≈0.68）。
ComfyUI 把 ControlNet hint 圖塞進畫布時**置中裁切**，切掉上下 → 頭頂與腳消失 → 只剩中段。
（與之前「全身 prompt 壓不過 CN」的假設不同；CN 一直忠實複製草圖，問題在 hint 圖被裁。）

### 方案（後端，主角色 + 變體）
- 移除上一輪的 `_apply_openpose_fullbody` 與骨架資產（使用者否決固定姿勢）。
- 新增 `_letterbox_to_aspect(image_bytes, w, h)` + `_border_color()`：
  全身模式把概念圖**補邊成畫布比例**（窄→左右補、寬→上下補），補邊用取樣背景色（與粉色背景無縫），
  置中貼上。如此 hint 圖與畫布同比例，ComfyUI 縮放為純等比、不裁切 → 全身（含草圖姿勢）完整保留。
- IPA 仍用**原始**概念圖（外觀），CN 用補邊後概念圖（結構/姿勢）。職責分離但都來自同一張草圖。
- `X-CN-Mode`：全身=`canny_fit`、表情特寫=`canny`。表情模式不補邊，維持原行為。

### 驗證
- 對真實 `concept_1`（303×691）letterbox 至 832×1216：輸出 473×691（AR 0.685 ≈ 畫布 0.684），
  目視確認全身置中、粉邊與背景無縫。
- 後端最大可編譯前綴含全部改動、無殘留 OpenPose 參照，`py_compile` 通過。

### 與低 IPA / 高 CN 的搭配
此法正好契合：姿勢與外觀都由 Canny（高權重 0.85+）從草圖取得，IPA 可保持低（≤0.4）。

### 可調項
若覺得補邊後人物偏小（側邊留白多），可改成「畫布比例直接跟隨草圖比例」讓人物填滿，
但極窄比例（<0.45）對 SDXL 稍偏離訓練分佈；目前先用 letterbox 穩定方案。

## 追加：歷史紀錄加入 IPA / CN 參數配置 ✅

### 需求
生圖歷史紀錄除模型外，需一併記錄 IPA 與 CN 的參數配置。

### 現狀盤點
- 歷史紀錄為前端 localStorage 功能（`App.jsx`，key `craftflow_history_v2`），各 Tab 透過 `onAddHistory` 寫入。
- 面板與 lightbox 已支援顯示 `item.model`，但**所有 Tab 實際都沒帶 `model`**（欄位閒置）。
- 人設圖（`CharacterTab.jsx`）的歷史項僅 `type/url/filename/label`。

### 改動（最小變更）
- `CharacterTab.jsx` 主角色與變體兩處 `onAddHistory`：
  - 補 `model`：取回應 header `X-Style`（checkpoint 風格）。
  - 補 `params`：`{ ipa, cn, cnMode }`。`ipa`/`cn` 為對應開關開啟時的權重值，關閉則 `null`；`cnMode` 取 header `X-CN-Mode`（`canny`/`canny_fit`）。
  - 參數值直接用前端送出的 state（免再解析），來源即實際生成所用設定。
- `App.jsx`：歷史面板與 lightbox 各加一行渲染 → `IPA 0.15 · CN 0.90 (canny_fit)`；關閉顯示 `IPA off`/`CN off`。
- `_saveHistory` 以 `{...h}` 展開保存，新欄位自動持久化，無需改動序列化。

### 備註
- 前端走 Vite proxy（`/api` → `:8000`，同源），`X-Style`/`X-CN-Mode` 等自訂 header 可正常讀取，後端 CORS 無需加 `expose_headers`。
- 後端 `X-Style`、`X-CN-Mode` 兩個 endpoint（主角色 + 變體）皆已輸出。

## OpenPose 後續（使用者已回覆，待節點安裝）
使用者回覆：
- preprocessor **未安裝 / 不確定** → 接線前必須先在 host 安裝，否則 ComfyUI 422 會讓全身生成全壞（違反 Resilient errors 鐵則），故暫不動現用 workflow。

### 當下解（零環境依賴，已可用）
